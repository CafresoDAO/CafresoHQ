# 2026-04-21 — Session 4: Deployment, NNS, first project

Fourth research session. Three goals, in priority order: close the **Deployment** branch of the MOC (which had been stubs since Session 1), parent the **Governance** branch by writing [[nns]] so the SNS notes stop dangling, and anchor the vault in a concrete target by specifying a first project. This also closes out the three open Session-2-deferred priorities ([[local-replica]], [[mainnet-deploy]], [[cycles-wallet]]) that Session 3 skipped when the user redirected to SNS.

## What was built

Four new Deployment concept notes, one Governance parent note, one end-to-end tutorial, and one project spec — all `status: draft` / `status: planning`, all qualitative where current prices/flags would otherwise drift:

- [[local-replica]] — `dfx start`, the local-network state directory, local canister IDs, and the places a single-node replica diverges from mainnet (consensus, subnet features, HTTP outcalls, randomness, clocks, cost). Cross-linked from [[dfx]] and [[mainnet-deploy]].
- [[mainnet-deploy]] — the full control-graph story of `dfx deploy --network ic`: mainnet identity, cycles funding path choice (cycles ledger vs. legacy wallet), management-canister `create_canister` + `install_code`, controllers, verification via `dfx canister --network ic status`. Explicit call-out that `caller` is the wallet principal under the legacy pattern.
- [[cycles-wallet]] — paired treatment of the legacy wallet canister and the current cycles-ledger canister. Covers ICP → cycles via the CMC (XDR-pegged), `dfx cycles` vs. `dfx wallet`, the three topping-up routes (cycles-ledger, wallet, CMC-direct), and flags DFINITY's current guidance to prefer the cycles ledger for new projects. Three open questions on exact flags and fees.
- [[asset-canister]] — the prebuilt `"type": "assets"` canister, `dfx.json` frontend entry shape, certified asset serving, `.ic-assets.json5` headers/fallback, SPA routing. Qualitative note on the `ic-frontend-canister` / `assets-api-v*` split (API layer separated from reference implementation).
- [[nns]] — the root DAO of the protocol. Covers neurons/following/voting, the NNS canister set (Governance, Registry, ICP Ledger, CMC, Root, Lifeline, Genesis Token) with per-canister roles, how adopted proposals execute through Root, and the precise relationship to [[sns]] (same shape, different scope — NNS pre-exists at genesis, SNS is minted by an NNS proposal, neurons do not cross over).
- [[ship-to-mainnet]] — a 45-minute, explicitly cautious end-to-end tutorial. Prefaced with a source-of-truth caveat: exact `dfx cycles convert` flags and the current XDR→cycles rate are not claimed; symbolic placeholders + links to the live docs. Includes a post-launch checklist and an explicit "set up a top-up plan" step.
- [[notes-dapp]] — a project *spec* (status: `planning`), not an implementation. Motoko backend + React + II + stable memory + asset canister. 10 milestones as checkboxes, with explicit non-goals and open design questions (stable-structure choice, per-user vs. shared canister, rate limiting). The prompt was explicit that this should be a spec only; no code to be written from it yet.
