# 2026-04-21 — Session 2: Patterns + Refinement

Second research session. Goal was to fill the Patterns layer of `icp-moc.md` — the four notes Session 1 flagged as priorities — and to revisit each Session 1 draft now that there is enough surrounding material to cross-link and verify against authoritative docs.

## New notes

Four atomic concept notes, all drafted from the Motoko docs in `dfinity/motoko` and the `dfinity/portal` docs:

- [[stable-memory]] — 500 GiB stable-memory ceiling; 64 KiB page / 128-page block model; Motoko `Region` API (`Region.new`, `Region.grow`, `Region.storeNat8`, …); Rust's `ic-stable-structures` with `MemoryManager`; how Enhanced Orthogonal Persistence changes the picture for Motoko.
- [[upgrade-hooks]] — Motoko `system func preupgrade() / postupgrade()` and Rust `#[pre_upgrade] / #[post_upgrade]`; explicit DFINITY guidance that hooks are **discouraged** under EOP; `persistent actor` keyword; `wasm_memory_persistence = opt keep` install option; the "brick on pre_upgrade trap" failure mode.
- [[inter-canister-calls]] — async + callback model; each `await` is a cross-message boundary with reentrancy implications; bounded-wait (best-effort) vs unbounded-wait (guaranteed) calls; attaching cycles; query methods cannot make inter-canister calls.
- [[candid]] — IDL + binary wire format; `.did` files; auto-generated in Motoko / `export_candid!()` in Rust; `candid:service` Wasm metadata; Candid subtyping and `didc check` for upgrade safety; `to_candid` / `from_candid` in Motoko (non-canonical, never use for hashing).

And one tutorial:

- [[counter-with-upgrade-hooks]] — a Motoko counter walked through three variants: v1 `persistent actor` (no hooks), v2 classical actor with `preupgrade` / `postupgrade` bridging a `transient HashMap`, and v3 "broken on purpose" showing the silent data-loss footgun. 25-minute exercise.

## MOC change

Added `candid` under **Development Stack** (not Patterns). Rationale: Candid is not a pattern developers opt into — it is the default interoperability layer the CDKs produce and consume, on par with `dfx`, `motoko`, and `rust-cdk`. Promoting it into Patterns would imply it's optional or situational, which it isn't. All five new notes (+ the renamed [[counter-with-upgrade-hooks]] tutorial) are marked with ✓; so is the previously-stub [[inter-canister-calls]].
