# 2026-04-21 — Session 5: SNS operations layer + 100M analysis

Fifth research session. Primary goal was operational: turn Session 3's five conceptual SNS notes into a complete, production-facing layer — pre-flight, tokenomics, testflight, post-launch operations, and a benchmarks table — then *use* that layer to evaluate a concrete user-proposed allocation.

## The user's question

> "I want to launch an SNS with 100M total supply: 15% dev, 30% swap, 55% treasury. Evaluate this."

Answered directly in [[tokenomics-100m-analysis]], the first-ever direct user-deliverable in the project layer of the vault. The analysis is framed not as "yes/no" on the headline split but as "the headline is defensible *at the extremes* but not *on the merits*, because every knob that actually controls governance (vesting, dissolve delays, minimum raise, NF match, per-participant caps) has not been specified".

### TL;DR for the user — before freezing the split, decide:

1. **Dev-bucket vesting + dissolve-delay schedule per named principal.** Single largest risk in the proposal — 15% with no schedule is a dump-risk allocation; 15% with multi-year vesting + multi-year dissolve delay is an alignment instrument.
2. **Treasury-use roadmap.** 55% treasury is *above* the commonly-observed 30–50% band; without a published roadmap for what the 55M tokens fund over what horizon, the community review will read the share as either oversized or unconsidered.
3. **Swap parameters (min raise, max raise, per-participant min/max, NF stance).** A 30% swap bucket is at the low end of the decentralization band; whether it produces real decentralization depends entirely on the minimum raise and whether the Neurons' Fund participates.
4. **Airdrop decision, explicit.** The proposal doesn't mention airdrops — that's a defensible choice but needs to be an *explicit* one, not silent.
5. **Model day-one voting-power distribution** under at least two dissolve-delay scenarios, and publish it. The community votes on *governance effect*, not on headline bucket percentages.

- [[sns-launch-readiness]] — pre-flight checklist. Dapp maturity, upgrade-hook hygiene (leans on [[upgrade-hooks]]), NNS-Root-as-co-controller handoff wiring, fallback controllers, audits, tokenomics review, community review, neuron / follow-graph planning.

Explicit on the distinction between vesting (transferability) and dissolve delay (both liquidity *and* voting weight).
- [[sns-testflight]] — dry-running the full launch flow against local replica or staging. Qualitative on the CLI surface because it drifts; explicit on what testflight proves (end-to-end wiring, swap math, neuron baskets, post-launch upgrade) and does not prove (mainnet cycle economics, NF behaviour, NNS adoption, legal/policy compliance).
- [[sns-operations]] — post-launch operating manual. Upgrading SNS canisters via SNS-W-blessed wasms, upgrading the dapp canister, nervous-system parameter changes, treasury disbursements, cycles and top-ups, emergency response patterns (no off-switch — pre-built halted wasm, feature-flag parameters, critical-proposal type), metrics + observability via dashboard.internetcomputer.org.
- [[sns-launch-benchmarks]] — the reference table for publicly-launched SNSs (OpenChat, Dragginz, Kinic, ICPSwap, BOOM DAO, WaterNeuron, Gold DAO, ELNA, YRAL). **Numeric cells left as `?`** with explicit instruction that they require verification from dashboard.internetcomputer.org/sns/<id> + the project's original proposal payload before citation. Qualitative landscape observations (typical bands) *are* captured from the public forum conversation — these are safe to assert without a per-project numeric claim.
- [[sns-end-to-end-launch]] — operational tutorial. 9 steps: community prep → NNS-Root-as-co-controller → draft `sns_init.yaml` → validate → testflight → submit proposal → swap → finalise → post-launch. Kept qualitative on CLI commands; explicit failure-mode section.
- [[tokenomics-100m-analysis]] — the direct user-deliverable. Benchmarks the 15/30/55 proposal against the qualitative band observations in [[sns-launch-benchmarks]], lists six ranked risks (treasury high, swap low, dev needs vesting, NF/airdrops/vesting gaps, voting-power model missing, ops cadence implicit), issues six concrete recommendations the user must resolve before freezing the split.
