# Token Utility and Value Capture Mechanisms

## The Value Triangle

Sustainable tokenomics requires three interconnected elements:
1. **Value Creation** - the product/protocol generates real economic value
2. **Value Accrual** - mechanisms funnel value toward the token
3. **Value Capture** - value is retained in the system, not extracted

## Core Utility Patterns

### Clear, Measurable Utility
- Users must understand what the token does and why it matters
- Avoid abstract "governance-only" tokens without operational use
- Best: tokens required for core protocol operations (gas, fees, access rights)

### Demand Drivers
Strong token demand comes from:
- **Fee requirements** - protocol charges in native token
- **Staking requirements** - lock tokens to participate/earn rewards
- **Access rights** - token-gated features or services
- **Burn mechanisms** - usage permanently removes supply

## Value Retention Mechanisms

Sustainable designs prevent value bleeding through:
- **Protocol fees** - margin on transactions flows to treasury/stakers
- **Protocol-owned assets** - treasury holds productive assets
- **Retained usage rights** - ongoing service requires ongoing token holding
- **Supply sinks** - buyback-and-burn, staking locks, LP requirements

## SNS Implications

For SNS init file design:
- Define **transaction fee** structure that feeds treasury ([[treasury-management-strategies]])
- Configure **neuron staking rewards** to incentivize long-term holding ([[neuron-staking-voting-power-mechanics]])
- Avoid pure inflation without offsetting demand drivers ([[token-supply-inflation-deflation-mechanisms]])
- Ensure governance participation has clear utility beyond voting ([[voting-rewards-and-incentive-design]])

## Anti-Patterns

- Tokens with no utility beyond speculation
- Uncapped inflation without demand sinks
- Value accrual flows entirely to early holders, not ongoing participants
- No mechanisms to retain value generated by protocol activity

## References
- Value capture requires structured mechanisms (fees, margins, protocol-owned assets)
- Volatile tokens and stable tokens serve different purposes - design for intended use case
- Interdisciplinary approach needed: economics, game theory, and protocol design
