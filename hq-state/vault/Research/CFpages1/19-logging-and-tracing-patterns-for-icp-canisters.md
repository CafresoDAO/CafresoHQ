# 19. Logging and Tracing Patterns for ICP Canisters

Effective logging is a cornerstone of [[18-monitoring-and-observability-for-canDB-canisters|canister observability]], providing the data needed for debugging, security auditing, and performance analysis. There are several patterns for logging on the Internet Computer, each suited for different needs.

### 1. Basic Canister Logs (`dfx`)

The most straightforward method for logging is using the `dfx canister logs` command. This command retrieves the standard output (e.g., from `Debug.print` in Motoko) generated by a canister during its execution. 

- **Use Case:** Primarily for local development and real-time debugging.
- **Limitations:** Logs are not persistently stored on-chain in a queryable way and can be difficult to manage in a production environment with multiple canisters.

### 2. Streaming Boundary Node Access Logs

For canisters exposed to the web, the Internet Computer allows for streaming access logs directly from the API Boundary Nodes. These logs provide visibility into the HTTP requests hitting your canister.

- **Use Case:** Analyzing web traffic, identifying request patterns, and detecting potential abuse or DoS attacks at the network edge.
- **Implementation:** Requires fetching the list of boundary nodes and setting up a log collection service to stream and aggregate these logs, as detailed in the official IC documentation.

### 3. Dedicated Logger Canister Pattern

A more robust and scalable solution is to implement a dedicated "Logger Canister." In this pattern, other canisters in the application send their log events to this central canister for storage and querying.

- **Use Case:** Creating a persistent, on-chain, and queryable audit trail or event log for the entire application.
- **Benefits:**
    - **Centralization:** All logs are in one place.
    - **Structured Data:** Logs can be sent in a standardized format (e.g., ICRC-3), making them easy to parse and query.
    - **Persistence:** Logs are stored in stable memory, surviving canister upgrades.
    - **Controlled Access:** You can define who is allowed to write logs and who can read them.
