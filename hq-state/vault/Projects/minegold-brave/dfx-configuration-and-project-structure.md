# DFX Configuration and Project Structure

The `dfx.json` file is the central configuration for the DFINITY Canister SDK, defining the project's canisters, their relationships, and network settings. It acts as a manifest for the entire application.

## Canister Definitions

The project is composed of three main canisters:

### 1. Backend Canister

This is a custom canister containing the core application logic. The configuration specifies the location of the compiled WebAssembly module and its Candid interface.

```json
"backend": {
  "type": "custom",
  "wasm": "src/backend/dist/backend.wasm",
  "candid": "src/backend/dist/backend.did",
  "build": "bash -c 'test -f src/backend/dist/backend.wasm && test -f src/backend/dist/backend.did || { echo \"[dfx.json] Pre-built backend artifacts missing.\"; echo \"         Run:  cd src/backend && mops install && mops build\"; exit 1; }'"
}
```

- **Type**: `custom` indicates it's a canister with compiled Wasm code.
- **Artifacts**: It points to the `backend.wasm` and `backend.did` files generated by the [[specialized-configuration-caffeine-and-mops|Mops build process]].
- **Build Command**: The `build` script doesn't compile the code itself. Instead, it verifies that the artifacts already exist, guiding the developer to run the correct build commands if they are missing. This is a key part of the [[verified-build-commands-and-dev-workflow|developer workflow]].

### 2. Frontend Canister

The frontend is a simple asset canister that serves the user interface.

```json
"frontend": {
  "type": "assets",
  "source": ["src/frontend/dist"],
  "dependencies": ["backend"]
}
```

- **Type**: `assets` tells `dfx` to serve static files from the specified `source` directory.
- **Dependencies**: Crucially, it depends on the `backend` canister. This ensures `dfx` deploys the backend first and makes its canister ID available to the frontend during its build/install process.

### 3. Internet Identity Canister

The project integrates with the official Internet Identity service for user authentication. The configuration pulls the necessary Wasm and Candid files from a remote URL.

```json
"internet_identity": {
  "type": "custom",
  "candid": "https://github.com/dfinity/internet-identity/releases/latest/download/internet_identity.did",
  "wasm": "https://github.com/dfinity/internet-identity/releases/latest/download/internet_identity_dev.wasm.gz",
  "remote": {
    "id": {
      "ic": "rdmx6-jaaaa-aaaaa-aaadq-cai"
    }
  }
}
```

This is a standard practice for using common services without needing to build them from source.

## Network Configuration

The file defines two primary networks:

- **`local`**: An `ephemeral` network for local development, binding to `127.0.0.1:4943`. Its state is reset every time the local replica is started.
- **`ic`**: The `persistent` mainnet network, configured to use a public provider at `https://icp-api.io`.

This separation allows for easy switching between local testing and [[mainnet-deployment-and-operational-playbook|mainnet deployment]].
